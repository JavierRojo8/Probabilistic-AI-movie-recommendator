────────────────────────────────────────────────────────────────────────────────
1.  BACKGROUND — THE TRAINED MODEL
────────────────────────────────────────────────────────────────────────────────

The core model is BPMF (Bayesian Probabilistic Matrix Factorisation). Instead
of learning a single point-estimate vector for each user and item, the model
learns a distribution over those vectors.

  Priors:
    U_i  ~  N(0, I)          user latent factor
    V_j  ~  N(0, I)          item latent factor
    b_u_i ~ N(0, 1)          user bias
    b_v_j ~ N(0, 1)          item bias

  Likelihood:
    R_ij ~ N( mu + b_u_i + b_v_j + U_i · V_j,  sigma_obs^2 )

Because exact Bayesian inference is intractable at scale, the posterior is
approximated by a mean-field Gaussian family (Variational Inference):

  q(U_i)   = N( mu_u[i],  diag(sigma_u[i]^2) )
  q(V_j)   = N( mu_v[j],  diag(sigma_v[j]^2) )
  q(b_u_i) = N( m_bu[i],  s_bu[i]^2 )
  q(b_v_j) = N( m_bv[j],  s_bv[j]^2 )

where sigma = exp(rho) enforces positivity via a log-parameterisation.

The variational parameters are trained by maximising the ELBO:

  ELBO = E_q[ log p(R | U, V, b_u, b_v) ]  -  KL( q || p )

The expected log-likelihood has a closed form because both the likelihood and
the variational family are Gaussian:

  E_q[ (r_ij - r_hat)^2 ]  =  (r_ij - E[r_hat])^2  +  Var[r_hat]

where the predictive variance decomposes as:

  Var[r_hat] = sigma_obs^2
             + Var[b_u_i] + Var[b_v_j]
             + sum_k( sigma_u_k^2 * sigma_v_k^2
                    + mu_u_k^2   * sigma_v_k^2
                    + mu_v_k^2   * sigma_u_k^2 )

This is the key quantity the model exposes as "uncertainty." It measures how
much the model's beliefs about a user-item pair are spread out.

Training uses Stochastic Variational Inference (SVI): the likelihood term is
computed on minibatches and scaled up by (n_total / batch_size) so the KL
terms remain in proportion. Best checkpoint: val RMSE = 0.8673 at epoch 5.


────────────────────────────────────────────────────────────────────────────────
2.  NEW USER ONLINE VARIATIONAL INFERENCE
────────────────────────────────────────────────────────────────────────────────

  THE PROBLEM
  -----------
  The trained model has fixed embedding tables of shape (n_users, K). A new
  user created during a live demo does not exist in those tables. We cannot
  simply assign user_idx = n_users + 1.

  THE SOLUTION
  ------------
  We treat the new user's variational parameters as free variables and optimise
  them against the same ELBO, holding all item parameters and sigma_obs frozen.

  Introduce fresh variational parameters for the new user:

    mu_u_new   in R^K         (latent factor mean)
    rho_u_new  in R^K         (latent factor log-std)
    m_bu_new   in R            (bias mean)
    rho_bu_new in R            (bias log-std)

  The per-user ELBO given their rated items {(j, r_j)} is:

    ELBO_new = sum_j E_q[ log p(r_j | U_new, V_j, b_u_new, b_v_j) ]
               - KL( q(U_new) || N(0, I) )
               - KL( q(b_u_new) || N(0, 1) )

  Crucially, there is NO minibatch scaling here. During training the likelihood
  is scaled by (n_total / batch_size) because each batch is a sample of the
  full dataset. For the new user we pass all of their ratings at once, so the
  sum already represents their complete evidence — no scaling is needed.

  The KL terms use the standard Gaussian formula:

    KL( N(mu, diag(sigma^2)) || N(0, I) )
      = 0.5 * sum_k( sigma_k^2 + mu_k^2 - 1 - 2*rho_k )

  We optimise with Adam (lr = 0.05, 200 steps). A higher learning rate than
  training (0.01) is appropriate because the optimisation problem is much
  smaller (4 scalar tensors of total size 2K+2 vs. millions of parameters).


────────────────────────────────────────────────────────────────────────────────
3.  SETBACK I — KL DOMINATION (poor cold-start personalisation)
────────────────────────────────────────────────────────────────────────────────

  THE PROBLEM
  -----------
  With K = 20 latent dimensions and N = 6 ratings, the ELBO decomposes as:

    ELBO  =  log_likelihood  (O(N) = O(6) terms)
           - KL              (O(K) = O(20) terms)

  At initialisation (mu_u = 0, rho_u = -3):

    KL_u  ≈  0.5 * K * 5  ≈  50      (per user factor vector)
    KL_bu ≈  0.5 * 5       ≈  2.5

  A typical log-likelihood contribution for one rating is roughly -1 to -2,
  so the total log-likelihood for 6 ratings is about -6 to -12. The KL term
  dominates by a factor of ~5x.

  This means the gradient from the KL ( = -mu_u, pulling toward zero) is
  much stronger than the gradient from the likelihood (pulling toward the
  user's actual taste). After 200 optimisation steps, mu_u barely moves from
  zero — which is the "average user." The result is that recommendations
  reflect globally popular movies (Shawshank Redemption, Schindler's List)
  rather than personal preferences.

  Observed symptom: a user who gave 5 stars to Toy Story, Toy Story 2, Beauty
  and the Beast, and Lion King, and 1 star to two drama films, received an
  entirely Drama-dominated top-10.

  OPTION CONSIDERED BUT REJECTED — KL ANNEALING (β-schedule)
  -----------------------------------------------------------
  One common fix is to weight the KL term by a factor β < 1:

    ELBO_β = log_likelihood - β * KL

  With β = 0.1, even 2-3 ratings can shift mu_u strongly. However this
  compromises calibration: the model becomes overconfident about the new
  user's preferences, and the predictive standard deviation (which drives
  the confidence bar in the UI) becomes artificially small. The uncertainty
  estimates would no longer be trustworthy. Rejected.

  THE SOLUTION — NEIGHBOUR WARM START
  ------------------------------------
  Rather than modifying the ELBO, we change the initialisation of mu_u and
  m_bu from the prior (zeros) to a similarity-weighted average of the learned
  latent vectors of the most similar training users.

  Step 1 — Inverted index (precomputed at startup):
    For every training rating (u, i, r), store i → [(u, r)].
    This allows O(1) lookup of all training users who rated each item.

  Step 2 — Candidate scoring:
    For each item j the new user rated with rating r_new_j:
      for each training user u who also rated j with r_train_j:
        record the squared difference (r_new_j - r_train_j)^2

    The mean squared difference across all co-rated items gives a similarity
    score per candidate user. Lower MSE = more similar taste.

  Step 3 — Weighted initialisation:
    Take the top-20 candidates (lowest MSE). Assign weights:

      w_k = 1 / (MSE_k + ε),   then normalise: w_k = w_k / sum(w)

    The warm-start mean vectors are:

      mu_u_init  = sum_k ( w_k * mu_u[neighbor_k] )
      m_bu_init  = sum_k ( w_k * m_bu[neighbor_k] )

  The Adam optimiser then starts in a region of latent space already close
  to the new user's true preferences. The KL gradient (pointing toward zero)
  and the likelihood gradient (pointing toward the neighbours' region) now
  act in opposition from a useful location, rather than the optimiser having
  to cross the full distance from zero.

  The rho parameters (log-stds) are still initialised at -3.0 (prior),
  meaning the model starts with low variance and tightens from there. This
  keeps the uncertainty estimates conservative while the mean converges.

4.  SETBACK II — TRAINING OVERFITTING (best epoch = 5 of 50)
────────────────────────────────────────────────────────────────────────────────

  THE PROBLEM
  -----------
  Observed training curve:

    Epoch  1:  val RMSE = 0.8932
    Epoch  5:  val RMSE = 0.8673  ← best checkpoint saved
    Epoch  6:  val RMSE = 0.8679  ← starts degrading
    Epoch 50:  val RMSE = 0.8850

  The model peaks at epoch 5 and then overfits for the remaining 45 epochs.
  The ELBO continues to improve throughout (training is "working"), but the
  val RMSE diverges — a classic generalisation failure.

  sigma_obs also drifts continuously downward (0.927 → 0.792), meaning the
  model becomes increasingly overconfident about its observation noise as it
  overfits to the training ratings.

  The KL regularisation from the variational prior is not strong enough to
  prevent the variational means (mu_u, mu_v) from memorising training ratings.

  CURRENT STATUS — PARTIALLY MITIGATED
  -------------------------------------
  The best-epoch checkpoint is saved and used for inference, so the deployed
  model is the epoch-5 version (val RMSE 0.8673). No damage in production.

  However, 45 epochs of compute are wasted. The recommended fix is KL
  annealing during training: start with a low KL weight (e.g. β = 0.1) and
  ramp it linearly to 1.0 over the first 10 epochs. This allows the likelihood
  to fit first before the prior exerts full regularisation pressure, which
  typically delays overfitting and yields a lower final val RMSE (~0.855-0.860
  estimated). This fix has not yet been applied.


────────────────────────────────────────────────────────────────────────────────
5.  SETBACK III — UNCERTAINTY THRESHOLD INCONSISTENCY
────────────────────────────────────────────────────────────────────────────────

  THE PROBLEM
  -----------
  The active-learning banner in the app was being triggered by two different
  comparisons that used the same threshold constant (1.2) but different
  quantities:

    app.py:          avg_unc = user_uncertainty() ** 0.5   → comparing STD to 1.2
    active_learning.py: user_is_uncertain()               → comparing VARIANCE to 1.2

  Since variance = std^2, this means the two checks triggered at very different
  levels of actual uncertainty. A user with std = 1.1 (variance = 1.21) would
  pass the active_learning.py check but fail the app.py check.

  THE SOLUTION
  ------------
  The app was updated to consistently use std (variance^0.5) for the displayed
  metric and for the threshold comparison. The active_learning.py function is
  a standalone utility whose threshold is independent. Both are now documented
  to make the units explicit.

6.  KNOWN REMAINING ISSUES
────────────────────────────────────────────────────────────────────────────────

  (a) NDCG METRIC INFLATION
      The NDCG@10 values reported (~0.87–0.92) are inflated because the
      ranking is computed only over each user's test items, not over all items.
      If a user has 3 test items and 2 of them are relevant, the model almost
      trivially achieves high NDCG. The absolute values are not comparable to
      published benchmarks. The metric remains valid for relative comparison
      between BPMF and SVD, but should not be cited as an absolute figure.

  (b) INGEST_DATA.PY SIDE EFFECT
      The download call download_movielens() runs at module level with no
      if __name__ == "__main__" guard. Any script that imports the module
      will silently trigger a network request. Low risk since no other module
      imports it, but should be fixed for robustness.

  (c) TRAINING KL ANNEALING NOT YET APPLIED
      As described in Setback II, the training loop would benefit from a
      β-annealing schedule on the KL term. Expected improvement: val RMSE
      from 0.867 to approximately 0.855–0.860, and a stable sigma_obs.

  (d) COMPATIBILITY / PERSONALISATION METRIC NOT YET IMPLEMENTED
      The current "Score" column shows E[r_ij] and the "Confidence" bar shows
      a normalised inverse-std. A more meaningful metric for the user is:

        P( r_ij >= threshold )  =  1 - Φ( (threshold - mu) / sigma )

      where Φ is the standard normal CDF and threshold is e.g. 4.0 stars.
      This gives the probability the user will genuinely enjoy the movie,
      coupling both the mean and the uncertainty into a single actionable
      number. Not yet implemented.
